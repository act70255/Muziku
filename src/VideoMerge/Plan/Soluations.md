# 圖片與多音檔合成長影片方案評估

## 目標

建立一套可自動化執行的工作流程，將：

- 一張靜態圖片
- 多個音訊檔案，實務上可先以 `m4a` 為主
- 可選的簡單視覺特效，例如雪花、粒子、慢速縮放

輸出為一支可直接上傳的長影片，重點是兼顧：

- 工具穩定性
- 輸出品質
- 處理時間
- 資源消耗
- 可維護性
- 後續擴展性

## 需求假設

### 輸入

- 一張主視覺圖片
- 多個音訊檔案，數量不固定，來源實務上可能多為 `m4a`
- 可選的循環特效影片或透明特效素材

### 輸出

- 單一長影片檔案，例如 `mp4`
- 影片長度等於所有音訊合併後的總長
- 畫面可為純靜態，或帶有輕微動態效果

### 非功能需求

- 能處理 `30 分鐘到 3 小時` 的長影片
- 不能因長時數而大量吃記憶體
- 能批次執行，未來可接資料夾監控或排程
- 日後可擴充字幕、片頭片尾、不同視覺模板

## 評估面向

### 工具面

- 是否容易安裝與部署
- 是否適合命令列自動化
- 是否容易整合到 Python 工作流

### 品質面

- 音訊合併是否穩定
- 長影片輸出是否容易出現不同步
- 視覺效果是否足夠自然
- 最終壓縮品質是否可控

### 處理時間面

- 是否接近線性處理
- 是否能避免整段影片全部載入記憶體
- 是否能在 CPU 環境下穩定輸出

### 擴展性面

- 是否容易加入更多素材前處理
- 是否容易加入規則，例如自動抓資料夾、檔名排序、輸出命名
- 是否容易延伸成批次產線或背景工作佇列

## 候選方案

### 方案 A：純 FFmpeg 指令自動化

#### 核心做法

- 使用 `ffprobe` 讀取音訊資訊
- 使用 `ffmpeg` 合併多個音訊
- 使用 `ffmpeg` 以單張圖片生成長影片
- 用濾鏡加入 `zoompan`、粒子疊加、淡入淡出等簡單視覺效果

#### 優點

- 效能最好，最適合長時數輸出
- 記憶體使用量低
- 工具成熟，跨平台穩定
- 很適合放入排程器、批次腳本或 CI 工作流

#### 缺點

- 指令複雜度高，可讀性較差
- 條件邏輯一多後不易維護
- 對錯誤處理、重試、輸出紀錄不夠友善

#### 品質評估

- 音訊與畫面同步性高
- 靜態圖搭配簡單視覺特效品質穩定
- 若特效需求升高，濾鏡腳本會迅速變難維護

#### 處理時間評估

- 在 `單圖 + 低 FPS + 簡單疊加` 情境下通常非常快
- 對 `60 分鐘` 級別輸出，通常明顯快於傳統剪輯軟體時間軸輸出
- 適合大量重複產出

#### 擴展性評估

- 中等
- 適合往更多濾鏡與命令模板擴充
- 不適合承載太多商業邏輯與流程控制

### 方案 B：Python 負責流程控制，FFmpeg 負責實際渲染

#### 核心做法

- 用 `Python` 掃描輸入資料夾、排序音檔、生成工作清單
- 用 `Python` 組出 `ffmpeg` 與 `ffprobe` 指令
- 用 `ffmpeg` 執行音訊標準化、拼接、影片輸出
- 用 `Python` 寫出日誌、設定檔、任務狀態與錯誤報告

#### 優點

- 保留 `FFmpeg` 的效能優勢
- 同時擁有 `Python` 的流程可讀性與擴展性
- 容易加入規則，例如檔名排序、音量正規化、自動命名輸出檔、失敗重試與批次處理

#### 缺點

- 比純 `FFmpeg` 多一層實作成本
- 需要定義清楚的設定格式與錯誤處理策略

#### 品質評估

- 輸出品質本質上與 `FFmpeg` 相同
- 但因流程更可控，較容易加入品質檢查
- 最適合做成可重複執行的穩定產線

#### 處理時間評估

- 實際渲染時間接近純 `FFmpeg`
- `Python` 帶來的額外成本通常可忽略
- 適合同時處理多個專案任務

#### 擴展性評估

- 高
- 最適合作為正式主線方案
- 後續可接字幕生成、多模板輸出、工作佇列、Web API 或桌面介面

### 方案 C：Python + MoviePy

#### 核心做法

- 直接用 `MoviePy` 載入圖片、音訊與特效素材
- 在 Python 內部完成合成與輸出

#### 優點

- 開發體驗直觀
- 程式碼表達力高
- 適合快速驗證原型

#### 缺點

- 長影片容易遇到速度慢、記憶體壓力高的問題
- 對 `1 小時以上` 內容不如 `FFmpeg` 穩定
- 實際底層仍高度依賴 `FFmpeg`

#### 品質評估

- 可做出正確結果
- 但在長片與批次生產情境下，不是最穩定選項

#### 處理時間評估

- 通常慢於直接使用 `FFmpeg`
- 越長的影片，差距越明顯

#### 擴展性評估

- 邏輯擴展容易
- 產線擴展普通
- 更適合原型與小規模工具，不適合作為長影片主渲染核心

### 方案 D：ComfyUI / AI 動畫流程

#### 核心做法

- 讓靜態圖經過 AI 節點生成局部動態
- 再與音訊組合輸出影片

#### 優點

- 視覺吸引力最高
- 可做出更有差異化的內容

#### 缺點

- 算力需求高
- 渲染時間長
- 流程複雜，穩定性較低
- 不適合先解決長影片量產需求

#### 品質評估

- 若調校得當，視覺效果最佳
- 但容易出現畫面漂移、閃爍、風格不穩等問題

#### 處理時間評估

- 最慢
- 不適合高頻批次生產

#### 擴展性評估

- 偏向創意擴展，不是產線擴展
- 適合作為後續加值方案，不適合作為 MVP 主線

## 方案比較總表

| 方案 | 工具複雜度 | 輸出品質穩定性 | 處理時間 | 資源消耗 | 自動化友善度 | 擴展性 | 建議定位 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `純 FFmpeg` | 中 | 高 | 快 | 低 | 高 | 中 | 高效生產核心 |
| `Python + FFmpeg` | 中高 | 高 | 快 | 低 | 很高 | 高 | 最推薦主線 |
| `MoviePy` | 低到中 | 中 | 中到慢 | 中到高 | 中 | 中 | 原型或小工具 |
| `ComfyUI` | 高 | 中 | 慢 | 很高 | 低 | 中 | 視覺加值實驗 |

## 推薦結論

### 最推薦方案

採用 `Python + FFmpeg` 的混合架構。

這個組合最符合目前需求，原因如下：

1. `FFmpeg` 可確保長影片輸出速度與穩定性
2. `Python` 可負責流程編排、設定管理、批次處理與錯誤控制
3. 後續若要加入字幕、片頭、模板、多專案排程，比純指令更容易維護
4. 不會像 `MoviePy` 一樣在長片情境下承受過多記憶體與效能壓力

### 不建議作為主線的方案

- 不建議以 `MoviePy` 作為長影片主渲染核心
- 不建議一開始就以 `ComfyUI` 當主要產線

### 可接受的 MVP 退階方案

若現在只想最快驗證可行性，可先做 `純 FFmpeg` 版本，等需求穩定後再包一層 `Python`。

## 建議技術架構

### 工具組合

- `FFmpeg`：核心轉碼、合併、濾鏡與輸出
- `FFprobe`：讀取音訊長度、編碼資訊、取樣率、聲道數
- `Python 3.12+`：任務編排與工作流控制
- `uv`：Python 環境與依賴管理

### Python 層建議職責

- 掃描輸入目錄
- 依檔名或設定排序音訊
- 驗證輸入檔案是否完整
- 產生 `ffmpeg` 所需的 manifest 或 filter 參數
- 執行命令並記錄 log
- 驗證輸出檔是否成功產生
- 產出 metadata，例如總時長、來源檔、輸出檔名、執行時間

### FFmpeg 層建議職責

- 音訊前處理
- 音訊串接
- 圖片轉影片
- 視覺特效疊加
- 最終封裝輸出

## 建議工作流程

### Step 1：掃描與驗證輸入素材

- 檢查圖片是否存在
- 檢查至少有一個音訊檔
- 讀取每個音訊的時長與格式，特別確認 `m4a` 的 codec、sample rate 與 channels
- 若音訊規格不同，先轉為統一格式，例如 `48k stereo`

### Step 2：音訊標準化

- 統一取樣率與聲道數
- 必要時做音量正規化，例如 `loudnorm`
- 若片段間差異過大，可加入極短淡入淡出

### Step 3：合併音訊

- 若來源規格完全一致，可用 `concat demuxer`
- 若來源規格不一致，先轉中介格式再拼接
- 最終輸出一條乾淨的主音軌

### Step 4：生成影片畫面

- 以單圖生成基礎畫面
- 加入可配置的輕微動態，例如 `Ken Burns` 慢速縮放、輕微平移、粒子或雪花循環覆蓋

### Step 5：合成與輸出

- 採 `1080p`
- 採 `H.264 + AAC`
- 若畫面大多靜態，可用 `10` 或 `12 FPS`
- 加上 `-tune stillimage`
- 視需求調整 `CRF`，例如 `20` 到 `24`

### Step 6：產出執行紀錄

- 任務開始與結束時間
- 使用了哪些輸入素材
- 輸出檔案大小
- 最終影片時長
- 失敗原因與錯誤訊息

## 品質控制建議

### 音訊品質

- 避免直接無條件 `-c:a copy`，除非所有來源參數完全一致
- 長片最怕片段之間音量落差過大
- 建議把音訊標準化納入正式流程，而不是交給人工碰運氣

### 視覺品質

- 靜態圖影片的重點不是高幀率，而是畫面穩定與壓縮效率
- 特效不要過多，否則只會增加渲染成本與壓縮雜訊
- 若使用黑底粒子素材，需確認混合模式與透明感是否自然

### 同步與輸出穩定性

- 輸出時長應以最終音訊為準
- 使用 `-shortest` 時要確認不會截斷應保留內容
- 每次輸出後至少驗證檔案可播放、時長正確、音訊存在，且首尾沒有黑畫面或靜音異常

## 若要開始實作

### 環境建議

- 使用 `uv init` 建立 Python 專案
- 使用 `uv add typer` 建立 CLI 入口
- 先不要引入 `MoviePy`，主流程以 `subprocess + ffmpeg` 為主

### 建議最小結構

- `src/VideoMerge/cli.py`：CLI 入口與參數解析
- `src/VideoMerge/application/workflow.py`：工作流程編排
- `src/VideoMerge/domain/models.py`：輸入設定與輸出結果模型
- `src/VideoMerge/infrastructure/ffmpeg_runner.py`：`ffmpeg` / `ffprobe` 呼叫封裝
- `src/VideoMerge/infrastructure/file_scanner.py`：掃描圖片、音訊與輸出路徑

### 第一個可交付版本建議

- 一個 CLI 指令，例如 `video-merge build --image cover.jpg --audio-dir music --output output.mp4`
- 一份 JSON 或 YAML 設定檔，用來定義 FPS、解析度、效果模板與輸出命名規則
- 一份執行結果報告，至少包含輸入檔、總時長、輸出路徑、處理耗時

## 處理時間判斷

### 對效能最有影響的因素

- 是否需要重新編碼音訊
- 是否加入多層視覺濾鏡
- 影片解析度與 FPS
- 特效素材是否需要額外縮放與混合

### 實務建議

- 若是 `單圖 + 輕微特效 + 1080p + 10 FPS`，`FFmpeg` 通常能在可接受時間內完成長片輸出
- 若影片主要用於 Lo-fi / Study 類型，沒有必要追求高 FPS
- 先用低成本參數做 MVP，再視實際觀看表現決定是否升級視覺複雜度

## 擴展方向

### 短期可加

- 自動讀取資料夾內所有音訊
- 依檔名排序
- 自動輸出標準命名檔案
- 多種視覺模板切換

### 中期可加

- 自動產生字幕或章節時間戳
- 自動加入片頭片尾
- 資料夾監控後自動觸發轉檔
- 批次任務佇列與失敗重跑

### 長期可加

- Web 管理介面
- 專案設定檔管理
- 雲端或多機分散處理
- AI 視覺增強作為可選流程分支

## MVP 實作建議

### 第一階段

先完成最小可行版本：

1. 讀取一張圖片
2. 讀取多個音訊檔
3. 合併成單一音軌
4. 生成 `1080p mp4`
5. 可選擇是否套用 `Ken Burns` 效果

### 第二階段

再補上：

1. 音量正規化
2. 粒子疊加模板
3. 自動命名與輸出資料夾結構
4. 失敗重試與 log

### 第三階段

最後再考慮：

1. 字幕
2. 章節
3. 片頭片尾
4. AI 動畫視覺增強

## 最終建議

若目標是 `穩定量產長影片`，最佳技術方向不是追求最炫的視覺，而是採用：

- `FFmpeg` 作為渲染核心
- `Python` 作為流程控制層
- `低 FPS + 靜態圖 + 輕微動態特效` 作為 MVP 視覺策略

這條路線在工具成熟度、品質穩定性、處理時間與可擴展性之間最平衡，也最適合後續逐步演進成真正可維護的自動化工作流。
